# Dual-Use Architecture

HPP V5 has two intended lives.

## 1. Buyer-Facing IP Asset

This layer must be clean, credible, and defensible.

Purpose:

- explain the post-LLM efficiency hypothesis
- demonstrate recurrent-depth architecture
- preserve evidence from previous HPP work
- show a clear product path
- protect patent-adjacent concepts
- make diligence easier for a serious buyer

Characteristics:

- polished docs
- small deterministic demos
- clean terminology
- explicit limitations
- evidence references
- reproducible measurements
- no private survival logs or operational memory

## 2. Private Operating Core

This layer serves the inventor and the field lab.

Purpose:

- run personal state/protocol loops
- coordinate shop work
- support project execution
- preserve field notes
- assist robotics planning
- become a practical nervous system for real-world operations

Characteristics:

- local-first memory
- offline operation
- battery-aware mode selection
- private logs
- flexible protocols
- robotics and shop adapters
- fast stop/resume behavior

## Shared Kernel

Both lives share the same conceptual kernel:

- recurrent loops
- Habit-14 stabilization
- state telemetry
- nurture/sentinel routing
- evidence discipline
- adaptive scaffolding

## Separation Rule

Buyer-facing artifacts may reference architecture, evidence, and demo outputs.

Private operating data stays out of buyer artifacts unless the inventor explicitly chooses to export or sanitize it.

## Near-Term Build Shape

The first HPP V5 app should contain:

- Today cockpit
- protocol run flow
- local evidence log
- Habit-14 progress
- power mode indicator
- architecture notes

The first HPP V5 model layer should contain:

- CUDA device diagnostics
- CPU fallback
- tiny inference harness
- prior-checkpoint inventory
- measurement logs
- clear power-mode guardrails
